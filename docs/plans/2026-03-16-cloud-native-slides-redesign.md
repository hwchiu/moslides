## 教學敘事弧線

整個課程的核心邏輯是「讓學生先感受分散式架構的痛，再看到 Container 的解法」：

1. **Part 1–2（架構複雜度上升）**：從單機到 15+ 台機器，每張圖比上一張更複雜，視覺壓力遞增
2. **Part 3（痛苦頂點 + 解法出現）**：Container 登場，複雜度驟降，視覺上要有「鬆一口氣」的感受
3. **Part 4–6（建立最佳實踐）**：12-Factor、DevOps、SRE，在簡化基礎上建立工程紀律

## 架構圖圖形語言

每種元件使用固定圖形，不可混用：

| 元件 | pptxgenjs 圖形 | 說明 |
|------|---------------|------|
| Client / Browser | `roundRect`（偏矮寬） | 代表終端使用者 |
| App Server | `roundRect`（標準比例） | 標準服務節點 |
| Load Balancer | `hexagon` | 代表分流交叉點 |
| Database | `cylinder`（pptx shape） | 業界標準 DB 圖示 |
| Cache (Redis) | `diamond` | 快速存取中間層 |
| Message Queue | `parallelogram` | 暗示排隊順序 |
| CDN / Internet | `cloud`（pptx shape） | 通用雲端標識 |
| Container / Pod | `roundRect`（虛線邊框） | 呼應貨櫃封裝意象 |

箭頭規格：實線箭頭（`→`）表示請求方向，虛線箭頭（`⤑`）表示 replication 或非同步流。

## 版面模板系統

### T1：封面頁（Slide 1）
- 全版深色背景
- 大標題置中，副標題較小
- 課程元資訊（時長、投影片數）在底部

### T2：Part 開場頁（Slide 2, 12, 20, 26, 34, 42 等）
- 左側大型 Part 編號（120pt，電藍半透明）
- 右側：Part 標題 + 本 Part 涵蓋項目 bullet

### T3：架構演進頁（核心頁型）
- 上 65%：架構圖（圖形語言系統）
- 下 35%：左欄 ✅ 優點，右欄 ⚠️ 問題（紅/綠分色卡片）

### T4：對比頁（❌ vs ✅）
- 雙欄並排
- 左欄紅色邊框「舊做法」
- 右欄綠色邊框「新做法」

### T5：概念說明頁
- 左 40%：大型圖示或數字
- 右 60%：3–4 點條列，每點有顏色 icon 圓圈

### T6：程式碼頁
- 深色代碼卡片（`#161B22` 背景）
- 關鍵字電藍，字串橘色，注釋灰色
- 右側說明文字

### T7：Part 小結頁
- 2×3 或 3×2 卡片格線
- 每卡片：概念名稱 + ✅/⚠️ 狀態 + 一行說明

### T8：複雜度進度條（出現在每張架構圖右上角）
- 長條形進度指示器
- 隨架構演進填色增加，Container 出現後遞減

## 執行計畫

分 6 個 Part 依序製作，每 Part 完成後產出截圖供確認：

| 批次 | 內容 | Slides |
|------|------|--------|
| Part 1 | 傳統部署演進 | 1–12 |
| Part 2 | Scale Out 挑戰 | 13–20 |
| Part 3 | Container 革命 | 21–26 |
| Part 4 | 12-Factor App | 27–34 |
| Part 5 | DevOps 整合 | 35–42 |
| Part 6 | SDLC 閉環 | 43–50 |

每批次流程：
1. 用 pptxgenjs 生成該 Part 的 slides
2. 轉換為圖片（LibreOffice + pdftoppm）
3. 視覺 QA（子 agent 檢查）
4. 修正問題後交使用者確認
5. 確認通過後繼續下一 Part

## 品質標準

- 所有架構圖必須使用圖形語言系統，禁止純文字 ASCII 箭頭
- 每頁至少一個視覺元素（圖形、圖示、卡片）
- 相同元件跨所有 slides 保持一致的顏色與圖形
- 箭頭必須標示方向與用途（用短標籤）
- 文字與背景對比度 ≥ 4.5:1
